# H2.4 — Muestras geolocalizadas

**Épica:** [E2 — Poblar el grafo](README.md)
**Requisito:** 2 · **Prioridad:** `[MUST]` · **Estado:** Pendiente
**Depende de:** H1.5, H2.1, H2.2, H2.3 · **Habilita:** H4.1, H6.2

## Historia de usuario
Como equipo, queremos varias muestras reales de distintas personas, en distintas provincias y fechas, cubriendo las tres ramas del protocolo, para que las consultas devuelvan datos coherentes y ricos.

## Criterios de aceptación
- [ ] Varias `cipa:Muestra` de **distintas personas**, en **distintas provincias** y **distintas fechas** (`cipa:fechaHora`).
- [ ] Cada muestra tiene `cipa:ejecutaProtocolo`, `cipa:respondidaPor`, `cipa:registradaEn` y una o más `cipa:MuestraRespuesta`.
- [ ] Cada `MuestraRespuesta` corresponde a un paso recorrido mediante `cipa:correspondeAPaso`.
- [ ] Las respuestas cubren valores de texto, opciones seleccionadas y adjuntos multimedia según el tipo de paso.
- [ ] Los adjuntos multimedia se modelan como `cipa:AdjuntoMultimedia` referenciado por `cipa:valorMultimedia`, no como parte del enunciado.
- [ ] Se cubren las **tres ramas** del protocolo de H2.1 (especie identificada / no identificada / cazuela vacía).
- [ ] **Multiplicidad simultánea:** ≥1 persona con muestras en 2 proyectos distintos.
- [ ] **Historial longitudinal:** al menos una persona con muestras en fechas separadas en el tiempo.

## Tareas
- [ ] Crear muestras cubriendo las 3 ramas.
- [ ] Variar provincia (GeoSPARQL/lat-long) y fecha.
- [ ] Asegurar el caso multi-proyecto y el caso longitudinal.

## Notas de modelado
- Esta data es la base de E4 (puntos), E5 (medallas) y casi todas las consultas de E6; conviene diseñarla pensando en los tres estados de medalla de [H4.3](../E4-participantes-jugando/H4.3-cobertura-demo.md).
- La compatibilidad entre `MuestraRespuesta` y el tipo de paso debe seguir la regla de H1.5: multimedia → `valorMultimedia`; cuestionario → `valorTexto` y/o `seleccionaOpcion`.
