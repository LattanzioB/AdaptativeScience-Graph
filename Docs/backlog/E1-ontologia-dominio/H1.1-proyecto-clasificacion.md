# H1.1 — Proyecto y clasificación

**Épica:** [E1 — Ontología de dominio](README.md)
**Requisito:** 1 · **Prioridad:** `[MUST]` · **Estado:** Pendiente
**Depende de:** H0.3 · **Habilita:** H1.2, H2.2

## Historia de usuario
Como modeladores, queremos representar un proyecto de ciencia participativa con su identificación, sus áreas de conocimiento y su protocolo activo, para que el proyecto sea la unidad central del dominio.

## Contexto (del PDF)
"Cada proyecto es la unidad central... Identificación: Nombre y descripción. Clasificación: Áreas del conocimiento (biología, astronomía, ecología). Protocolo: Cada proyecto posee un único protocolo activo."

## Criterios de aceptación
- [ ] `cipa:Project` con `rdfs:label`/`dct:description` (nombre y descripción del objetivo).
- [ ] Áreas de conocimiento modeladas como `skos:Concept` dentro de un `skos:ConceptScheme` (`cipa:KnowledgeAreaScheme`).
- [ ] Propiedad `cipa:hasArea` (Project → skos:Concept), N–N.
- [ ] Relación a **un** protocolo activo: `cipa:hasActiveProtocol` (cardinalidad funcional / `owl:maxCardinality 1`).

## Tareas
- [ ] Declarar clases y propiedades.
- [ ] Crear el esquema SKOS de áreas (biología, ecología, astronomía, …).
- [ ] Documentar la restricción de "único protocolo activo".

## Notas de modelado
- "Único protocolo **activo**" ≠ "único protocolo": un proyecto puede tener históricos inactivos pero un solo activo. Modelar con `hasActiveProtocol` funcional + `hasProtocol` general.
