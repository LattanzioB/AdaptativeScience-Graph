# H5.1 — Patrón `sh:sparql` "conformidad = ganador"

**Épica:** [E5 — Shapes de medallas](README.md)
**Requisito:** 5 · **Prioridad:** `[MUST]` · **Estado:** Pendiente
**Depende de:** H0.4, H4.3 · **Habilita:** H5.2

## Historia de usuario
Como equipo, queremos fijar el patrón SHACL con el que una shape determina a la persona ganadora de una medalla, para que E5 sea consistente y reproducible.

## Decisión de modelado (16/06)
**`sh:sparql` dentro de un `sh:NodeShape` por medalla**, con la convención **conformidad = ganador**.

## Contexto (del PDF)
Req. 5: "determinar si una persona es ganadora de la medalla a través de Shapes."

## Criterios de aceptación
- [ ] Un `sh:NodeShape` por medalla que **targetea** `cipa:Persona` (`sh:targetClass`).
- [ ] El criterio se expresa con una restricción `sh:sparql` cuyo `SELECT` devuelve `$this` como **violación** cuando la persona **NO** cumple (umbral de puntos / cantidad de muestras o respuestas / medalla previa).
- [ ] Por lo tanto: la persona que **conforma** (sin violaciones) es **ganadora**; la que no cumple genera un `sh:ValidationResult`.
- [ ] La inversión lógica (el SELECT devuelve a quienes *no* ganan) queda **documentada** con un ejemplo.

## Tareas
- [ ] Escribir un shape de prueba para M1 y validar la inversión con un caso ganador y uno no ganador.
- [ ] Documentar el patrón en `docs/decisiones-modelado.md`.

## Notas de modelado
- Alternativa considerada y descartada: "conformidad = ganador" con restricciones nativas (`sh:minCount`/`sh:qualifiedValueShape`) sobre un path derivado. Se eligió `sh:sparql` por flexibilidad para umbrales y precedencia.
- Verificar que el motor SHACL (pyshacl / RDF4J) soporte `sh:sparql` (ver [H0.4](../E0-infraestructura/H0.4-toolchain-validacion.md)).
- El target cambia con la reestructuración: el actor del dominio es `cipa:Persona`.
