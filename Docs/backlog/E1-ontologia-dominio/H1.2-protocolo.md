# H1.2 — Protocolo

**Épica:** [E1 — Ontología de dominio](README.md)
**Requisito:** 1 · **Prioridad:** `[MUST]` · **Estado:** Pendiente
**Depende de:** H1.1 · **Habilita:** H1.3, H1.4

## Historia de usuario
Como modeladores, queremos representar el protocolo de recolección con su estado y su composición por pasos, para definir cómo se capturan los datos.

## Contexto (del PDF)
"El Protocolo... Estado: activo o inactivo. Composición: serie de pasos secuenciales. Orden: lógica cronológica o procedimental."

## Criterios de aceptación
- [ ] `cipa:Protocol rdfs:subClassOf sosa:Procedure`.
- [ ] Propiedad `cipa:status` con valores controlados (`active` / `inactive`) — vía `skos`/individuos o `xsd:string` enumerado.
- [ ] Composición por pasos: `cipa:hasStep` (Protocol → Step).
- [ ] El orden de los pasos se modela con transiciones (ver [H1.4](H1.4-transiciones.md)), no con un índice plano, para soportar bifurcación.

## Tareas
- [ ] Declarar `Protocol` como subclase de `sosa:Procedure`.
- [ ] Modelar `status` y marcar la relación con `Project` (inversa de `hasActiveProtocol`).

## Notas de modelado
- Reutilizar `sosa:Procedure` alinea el protocolo con el muestreo SOSA de [H1.5](H1.5-muestreo.md).
