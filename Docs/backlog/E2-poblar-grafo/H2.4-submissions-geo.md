# H2.4 — Submissions geolocalizadas

**Épica:** [E2 — Poblar el grafo](README.md)
**Requisito:** 2 · **Prioridad:** `[MUST]` · **Estado:** Pendiente
**Depende de:** H1.5, H2.1, H2.2, H2.3 · **Habilita:** H4.1, H6.2

## Historia de usuario
Como equipo, queremos varias muestras reales de distintos participantes, en distintas provincias y fechas, cubriendo las tres ramas del protocolo, para que las consultas devuelvan datos coherentes y ricos.

## Criterios de aceptación
- [ ] Varias `Submission` de **distintos participantes**, en **distintas provincias** y **distintas fechas** (`resultTime`).
- [ ] Cada submission con observaciones por paso (geo, hora, archivo, opciones, estado, texto).
- [ ] Se cubren las **tres ramas** del protocolo de H2.1 (especie identificada / no identificada / cazuela vacía).
- [ ] **Multiplicidad simultánea:** ≥1 participante con submissions en 2 proyectos distintos.
- [ ] **Historial longitudinal:** al menos un participante con submissions en fechas separadas en el tiempo.

## Tareas
- [ ] Crear submissions cubriendo las 3 ramas.
- [ ] Variar provincia (GeoSPARQL/lat-long) y fecha.
- [ ] Asegurar el caso multi-proyecto y el caso longitudinal.

## Notas de modelado
- Esta data es la base de E4 (puntos), E5 (medallas) y casi todas las consultas de E6; conviene diseñarla pensando en los tres estados de medalla de [H4.3](../E4-participantes-jugando/H4.3-cobertura-demo.md).
