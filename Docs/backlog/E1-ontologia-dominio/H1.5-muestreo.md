# H1.5 — Muestreo (SOSA + PROV)

**Épica:** [E1 — Ontología de dominio](README.md)
**Requisito:** 1 · **Prioridad:** `[MUST]` · **Estado:** Pendiente
**Depende de:** H1.3, H1.6 · **Habilita:** H2.4, H4.1

## Historia de usuario
Como modeladores, queremos representar cada aporte de un usuario como una colección de observaciones atadas a los pasos del protocolo, para registrar las respuestas con su atribución permanente.

## Contexto (del PDF)
"Cada vez que una persona realiza un aporte se genera una instancia de respuesta... Ubicación, Hora, Archivo, Datos." "Atribución de Datos: cada muestra queda vinculada permanentemente al usuario que la generó."

## Criterios de aceptación
- [ ] `cipa:Submission rdfs:subClassOf sosa:ObservationCollection`, vinculada al `Protocol` respondido.
- [ ] Cada dato por paso es un `sosa:Observation` con `sosa:resultTime` (hora), `sosa:hasFeatureOfInterest` y resultado tipado (geo / archivo / opción seleccionada / texto / estado), referenciando el `Step` que responde.
- [ ] Atribución permanente: `cipa:submittedBy rdfs:subPropertyOf prov:wasAttributedTo` (Submission → Participant).
- [ ] La ubicación usa GeoSPARQL (`geo:hasGeometry`/`geo:asWKT`) o lat/long; coherente con H0.2/H2.4.

## Tareas
- [ ] Modelar `Submission` y `Observation` por paso.
- [ ] Declarar `submittedBy` como subpropiedad de `prov:wasAttributedTo` (clave para H6.3).
- [ ] Definir el tipado de resultados según tipo de paso.

## Notas de modelado
- Usar `prov:wasAttributedTo` como superpropiedad habilita la **consulta de inferencia** de [H6.3](../E6-graphdb-consultas/H6.3-consulta-inferencias.md).
- `resultTime` por observación habilita el **historial longitudinal** del PDF.
