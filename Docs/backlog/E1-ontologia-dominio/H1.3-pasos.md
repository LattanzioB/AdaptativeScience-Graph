# H1.3 — Pasos (jerarquía de tipos)

**Épica:** [E1 — Ontología de dominio](README.md)
**Requisito:** 1 · **Prioridad:** `[MUST]` · **Estado:** Completada
**Depende de:** H1.2 · **Habilita:** H1.4, H1.5, H2.1

## Historia de usuario
Como modeladores, queremos representar los pasos del protocolo y sus enunciados, para distinguir entre contenido multimedia, preguntas y opciones de respuesta sin mezclar definición del protocolo con respuestas de una muestra.

## Contexto (del PDF + diagrama)
El modelo reestructurado define `Paso` como clase base y sólo especializa los tipos que cambian la definición del enunciado: `PasoMultimedia` y `PasoCuestionario`. La ubicación, fecha, texto respondido y adjuntos pertenecen a la ejecución (`Muestra`/`MuestraRespuesta`), no a nuevos tipos de paso.

## Criterios de aceptación
- [x] `cipa:Paso` como clase base.
- [x] `cipa:PasoMultimedia rdfs:subClassOf cipa:Paso`.
- [x] `cipa:PasoCuestionario rdfs:subClassOf cipa:Paso`.
- [x] `cipa:Paso` se vincula con `cipa:Enunciado` mediante `cipa:tieneEnunciado`.
- [x] `cipa:PasoMultimedia` usa `cipa:EnunciadoMultimedia`; **no** contiene el archivo subido por la persona.
- [x] `cipa:PasoCuestionario` usa `cipa:EnunciadoCuestionario`, `cipa:TipoRespuesta` y `cipa:OpcionRespuestaCuestionario`.
- [x] `TipoRespuesta` cubre texto libre, selector único y selector múltiple.

## Tareas

- [x] Documentar que el paso inicial vive en `Protocolo.tienePrimerPaso` (H1.2) y que el cierre del flujo se representa como un paso terminal del protocolo.

## Notas de modelado
- Un texto libre o un checkbox se modelan como `PasoCuestionario` con distinto `TipoRespuesta`.
- El adjunto multimedia subido por la persona se modela en `MuestraRespuesta.valorMultimedia` (H1.5), no en `EnunciadoMultimedia`.
