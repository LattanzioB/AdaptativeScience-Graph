# H1.2 — Protocolo

**Épica:** [E1 — Ontología de dominio](README.md)
**Requisito:** 1 · **Prioridad:** `[MUST]` · **Estado:** Completada
**Depende de:** H1.1 · **Habilita:** H1.3, H1.4

## Historia de usuario
Como modeladores, queremos representar el protocolo de recolección con su estado, su composición por pasos y su paso inicial, para definir cómo comienza y avanza la captura de datos.

## Contexto (del PDF)
"El Protocolo... Estado: activo o inactivo. Composición: serie de pasos secuenciales. Orden: lógica cronológica o procedimental."

## Criterios de aceptación
- [x] `cipa:Protocolo` con `cipa:tieneNombre`, `cipa:tieneDescripcion` y `cipa:tieneEstado`.
- [x] `cipa:EstadoProtocolo` controla al menos los estados activo/inactivo.
- [x] Composición por pasos: `cipa:tienePaso` (Protocolo → Paso).
- [x] Paso inicial explícito: `cipa:tienePrimerPaso` (Protocolo → Paso), obligatorio para ejecutar el flujo.
- [x] El orden de los pasos se modela con transiciones (ver [H1.4](H1.4-transiciones.md)), no con un índice plano, para soportar bifurcación.

## Tareas
- [x] Declarar `Protocolo` y sus propiedades descriptivas.
- [x] Modelar `EstadoProtocolo`.
- [x] Marcar la relación con `Proyecto` mediante `tieneProtocolo` y `tieneProtocoloActivo`.

## Notas de modelado
- `Protocolo` puede alinearse con `sosa:Procedure` en [H1.7](H1.7-reutilizacion-alineaciones.md), pero la estructura propia del dominio queda expresada con `tienePaso` y `tienePrimerPaso`.
