# H1.9 — Perfil del Colaborador

**Épica:** [E1 — Ontología de dominio](README.md)
**Requisito:** 1 (módulo "Gestión de Participantes y Perfiles" del PDF) · **Prioridad:** `[SHOULD]` · **Estado:** Pendiente
**Depende de:** H1.5, H1.6 · **Habilita:** H6.2

> **Por qué existe esta historia:** el PDF describe un "Perfil del Colaborador" (intereses, expertise, engagement) que **no** aparece entre los entregables 1–6. Se incluye liviano para que la ontología "modele el dominio" completo (Req. 1) sin sobre-invertir. Decisión del 16/06.

## Historia de usuario
Como modeladores, queremos que el perfil del colaborador (intereses y nivel de expertise) sea **derivable** de su actividad, para describir el dominio de perfiles sin duplicar datos.

## Contexto (del PDF)
"Perfil de Usuario dinámico: Análisis de Intereses (áreas/protocolos donde es más activo), Nivel de Expertise (calidad/frecuencia de aportes), Interacción Personalizada."

## Criterios de aceptación
- [ ] **Intereses derivables:** las áreas de interés de un participante se obtienen vía `submission → protocol → project → area` (no se aseveran a mano). Se documenta la consulta/derivación.
- [ ] **Nivel de expertise:** atributo `cipa:expertiseLevel` (p. ej. `skos:Concept` novato/intermedio/experto) asignable según frecuencia de aportes; se modela la clase/propiedad aunque el cálculo sea manual o por regla.
- [ ] El perfil **no** introduce datos redundantes: se apoya en submissions ya existentes (H1.5).

## Tareas
- [ ] Declarar `cipa:expertiseLevel` y su escala.
- [ ] Documentar la derivación de intereses (alimenta [H6.2](../E6-graphdb-consultas/H6.2-consultas-multihop.md)).
- [ ] (Opcional) `cipa:interestArea` como propiedad derivada/inferible.

## Notas de modelado
- "Análisis de intereses" se realiza como una de las consultas multi-hop de dominio (H6.2), cerrando el módulo de perfiles con un entregable concreto.

## Artefactos
- `ontology/cipa-domain.ttl` (expertiseLevel, interestArea)
