# H1.6 — Persona participante

**Épica:** [E1 — Ontología de dominio](README.md)
**Requisito:** 1 · **Prioridad:** `[MUST]` · **Estado:** Completada
**Depende de:** H0.3 · **Habilita:** H1.5, H1.9, H4.2

## Historia de usuario
Como modeladores, queremos representar a la persona que responde muestras con identidad única, para rastrear sus aportes de manera unívoca.

## Contexto (del PDF)
"El Participante... Identidad Única: registro que permite rastrear sus aportes de manera unívoca." "Multiplicidad Simultánea: una persona puede colaborar en diversos proyectos al mismo tiempo."

## Criterios de aceptación
- [x] `cipa:Persona` representa a quien responde o genera una `Muestra`; puede alinearse con `foaf:Person` y `prov:Agent`.
- [x] Identidad única (`cipa:personaId` con `owl:hasKey` o una propiedad funcional inversa si se usa email).
- [x] La relación N–N persona↔proyecto **emerge de las muestras** (no se asevera directamente); opcionalmente una propiedad derivada `cipa:participaEn`.

## Tareas
- [x] Declarar `Persona` y sus posibles superclases.
- [x] Definir la clave de identidad única.
- [x] Documentar cómo se deriva la participación en proyectos desde las muestras.

## Notas de modelado
- Modelar la participación como **emergente** evita inconsistencias y habilita la consulta multi-hop de [H6.2](../E6-graphdb-consultas/H6.2-consultas-multihop.md): `Persona ← respondidaPor ← Muestra → Protocolo → Proyecto → AreaConocimiento`.
