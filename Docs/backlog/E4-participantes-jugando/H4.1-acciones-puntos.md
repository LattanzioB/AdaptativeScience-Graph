# H4.1 — Acciones → eventos de puntos

**Épica:** [E4 — Participantes jugando](README.md)
**Requisito:** 4 · **Prioridad:** `[MUST]` · **Estado:** Completado
**Depende de:** H2.4, H3.3 · **Habilita:** H4.2, H5.2

## Historia de usuario
Como equipo, queremos que cada acción relevante de una persona genere un evento de puntos según la regla correspondiente, para conectar el grafo de dominio con el juego.

## Contexto (del PDF)
Req. 4: "los participantes están jugando al juego definido en 3."

## Criterios de aceptación
- [x] Cada `cipa:Muestra` o `cipa:MuestraRespuesta` relevante (de E2) genera un `pbl:PointAward` según la `pbl:PointRule` aplicable (de H3.3).
- [x] El `pbl:PointAward` referencia la acción de dominio que lo originó vía `pbl:fromContribution` (muestra completa, respuesta con adjunto multimedia) para mantener trazabilidad dominio→juego.
- [x] Las acciones administrativas (editar perfil) **no** generan `pbl:PointAward`.

## Tareas
- [x] Mapear cada tipo de acción de E2 a su regla de H3.3 (protocolo terminal=50, `valorMultimedia`=20, geolocalización=10).
- [x] Instanciar los `pbl:PointAward` correspondientes en `data/juego-eventos.ttl`.

## Notas de modelado
- Este es el **punto de acople** entre `cipa:` y `pbl:`; las ontologías quedan separadas (E1/E3) y se unen acá vía la capa de datos.
- Se conservan las IRIs existentes: dominio en `cipa:`, ludificación en `pbl:` y datos concretos en `cipa-data:`.
- Acople persona↔jugador: cada `pbl:Player` se enlaza a su `cipa:Persona` con `owl:sameAs`.
- Trazabilidad: se agregó `pbl:fromContribution` (en `ontology/pbl.ttl`, sin rango para no acoplar la T-Box) que liga el award con la `cipa:Muestra`/`cipa:MuestraRespuesta` de origen.
- "Protocolo completado" = la muestra tiene respuesta para un paso **terminal** (sin transición saliente): `foto-arbol`/`confirmar-exotica` (árboles) o `aves-cantidad` (aves).
