# H1.1 — Proyecto y área de conocimiento

**Épica:** [E1 — Ontología de dominio](README.md)
**Requisito:** 1 · **Prioridad:** `[MUST]` · **Estado:** Pendiente
**Depende de:** H0.3 · **Habilita:** H1.2, H2.2

## Historia de usuario
Como modeladores, queremos representar un proyecto de ciencia participativa con su identificación, su área de conocimiento y su protocolo activo, para que el proyecto sea la unidad central del dominio reestructurado.

## Contexto (del PDF)
"Cada proyecto es la unidad central... Identificación: Nombre y descripción. Clasificación: Áreas del conocimiento (biología, astronomía, ecología). Protocolo: Cada proyecto posee un único protocolo activo."

## Criterios de aceptación
- [ ] `cipa:Proyecto` con `cipa:tieneNombre` y `cipa:tieneDescripcion`.
- [ ] `cipa:AreaConocimiento` modela la clasificación temática del proyecto; puede alinearse con `skos:Concept` dentro de `cipa:EsquemaAreasConocimiento`.
- [ ] Propiedad principal `cipa:perteneceA` (Proyecto → AreaConocimiento) y, si se necesita la navegación inversa, `cipa:clasificaProyecto` (AreaConocimiento → Proyecto).
- [ ] Relación a **un** protocolo activo: `cipa:tieneProtocoloActivo` (cardinalidad funcional / `owl:maxCardinality 1`).
- [ ] Relación general `cipa:tieneProtocolo` para protocolos históricos o inactivos.

## Tareas
- [ ] Declarar clases y propiedades.
- [ ] Crear el esquema de áreas (biología, ambiente, botánica, zoología, …).
- [ ] Documentar la restricción de "un protocolo activo" sin impedir varios protocolos históricos.

## Notas de modelado
- "Único protocolo **activo**" ≠ "único protocolo": un proyecto puede tener históricos inactivos pero un solo activo. Modelar con `tieneProtocoloActivo` funcional + `tieneProtocolo` general.
